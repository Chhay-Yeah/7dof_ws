# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workspace

7-DOF robotic arm workspace with Gazebo/Ignition simulation, ros2_control, and MoveIt2 motion planning.

**Packages**:
- `arm_bot` — robot description (URDF/Xacro), launch files, controllers config, FK/IK Python nodes, C++ Cartesian demo
- `arm_moveit_config` — MoveIt2 config generated by Setup Assistant (SRDF, planners, kinematics, controller mappings)

## Key Commands

```bash
# Source workspace (always do this first)
source ~/7dof_ws/install/setup.bash

# Build
cd ~/7dof_ws
colcon build
colcon build --packages-select arm_bot
colcon build --packages-select arm_moveit_config

# Run simulation (run in order, separate terminals)
ros2 launch arm_bot gazebo.launch.py                  # Start Gazebo + spawn robot
ros2 launch arm_bot bot_controller.launch.py          # Start ros2_control + controllers
ros2 launch arm_bot rviz.launch.py                    # Visualize in RViz

# MoveIt demo (no real robot — fake hardware)
ros2 launch arm_moveit_config demo.launch.py

# MoveIt with Gazebo
ros2 launch arm_moveit_config moveit.launch.py
ros2 launch arm_moveit_config moveit_rviz.launch.py

# Individual nodes
ros2 run arm_bot fk_arm_v3.py           # URDF-driven FK (subscribes /robot_description)
ros2 run arm_bot ik_arm_v3.py           # URDF-driven DLS IK with null-space centering
ros2 run arm_bot ik_to_trajectory.py    # IK output → JointTrajectory for arm_controller
ros2 run arm_bot cartesian_path.py      # Generates Cartesian /ee_target sequences
ros2 run arm_bot slider_controller.py
ros2 run arm_bot drawing_trajectory_planner.py
ros2 run arm_bot moveit_client.py
ros2 run arm_bot relay_node.py          # Relay /joint_commands → /joint_states (no real HW)

# C++ demos (after MoveIt is running)
ros2 run arm_bot arm_bot                # move_program.cpp — Cartesian linear + circular paths
ros2 run arm_bot cartesian_move         # cartisian_move.cpp — alternate Cartesian demo

# Testing / IK visualization
ros2 launch arm_bot fk_rviz_test.launch.py    # RViz + robot_state_publisher + joint_state_publisher + fk_arm_v3
ros2 launch arm_bot ik_rvize_test.launch.py   # RViz + relay_node + ik node (no Gazebo)
ros2 launch arm_bot ik_gazebo.launch.py       # IK node driving Gazebo end-to-end
ros2 launch arm_bot rviz_joint_gui.launch.py  # RViz + joint_state_publisher_gui (URDF inspection)
ros2 launch arm_bot arm_slider.launch.py      # bot_controller + GUI slider + slider_controller
```

## Architecture

### Control Pipeline

```
[Gazebo/Ignition] ←→ [ign_ros2_control plugin]
                            ↕
                   [ros2_control manager]  @ 100 Hz
                            ↕
              [JointTrajectoryController: arm_controller]
                            ↕
              [JointStateBroadcaster → /joint_states]
```

### Motion Control Modes

**1. Direct joint commands (GUI)**
`arm_slider.launch.py` → `slider_controller.py` converts `JointState` → `JointTrajectory` → `arm_controller`

**2. Custom Jacobian IK (Python — `ik_arm_v3.py`)**
- Builds FK + 6×N geometric Jacobian directly from `/robot_description` (latched URDF) — no DH idealisation, exactly matches `robot_state_publisher`/RViz.
- Subscribes `/joint_states` + `/ee_target`; publishes `/joint_commands` at 200 Hz.
- Damped Least Squares with **adaptive damping** (`λ` shrinks from 0.05 → 5e-4 as the residual closes), 4 inner iterations per tick.
- Null-space joint-centering toward URDF mid-limits (`NULL_K=0.3`).
- Per-step `‖dq‖` cap at 0.10 rad to suppress singularity overshoot.
- Tolerances: 10 µm position, ~0.006° orientation.
- Parameters: `base_link` (default `base_link`), `tip_link` (default `ee` — override to `j7_Link` to match this robot).

**3. URDF-driven FK (`fk_arm_v3.py`)**
- Same URDF chain as `ik_arm_v3.py`; publishes `/ee_pose` from `/joint_states`.
- Source of truth for end-effector pose when comparing against MoveIt or RViz.
- Same `base_link` / `tip_link` parameters as the IK node.

**4. IK testing without Gazebo (`ik_rvize_test.launch.py`)**
- Launches RViz + `relay_node.py` + the IK node.
- `relay_node.py` (`JointRelay`): subscribes to `/joint_commands`, republishes as `/joint_states` at 50 Hz — simulates hardware feedback loop without a simulator.

**5. MoveIt2 motion planning**
- `moveit_client.py` bridges `/ee_target` → MoveIt `MoveGroup` action
- Planner: OMPL (default), also configured: CHOMP, TrajOpt, Pilz, LERP
- IK solver: KDL, timeout 0.005 s, resolution 0.005 rad
- Default velocity/acceleration scaling: 0.1

**6. C++ Cartesian paths (`move_program.cpp` → `arm_bot`, `cartisian_move.cpp` → `cartesian_move`)**
- Uses `MoveGroupInterface::computeCartesianPath()`
- Linear and circular paths; requires >90% feasibility before executing
- Publishes waypoint markers to RViz
- Note the source filename misspelling `cartisian_move.cpp` (executable target is `cartesian_move`).

### Key Topics

| Topic | Type | Direction |
|---|---|---|
| `/joint_states` | `sensor_msgs/JointState` | hardware → nodes |
| `/joint_commands` | `sensor_msgs/JointState` | FK/IK nodes → controller |
| `/ee_pose` | `geometry_msgs/PoseStamped` | FK node output |
| `/ee_target` | `geometry_msgs/PoseStamped` | target for IK / MoveIt |
| `/arm_controller/joint_trajectory` | `trajectory_msgs/JointTrajectory` | trajectory input |

### Robot Kinematics

7 revolute joints: `joint_1`–`joint_7`.
Links: `base_link`, `Link_1`–`Link_7`, `ee` (end-effector frame).
**Tip link for FK/IK nodes is `ee`** — `fk_arm_v3.py`/`ik_arm_v3.py` default to `ee` and `fk_rviz_test.launch.py` passes `tip_link:=ee`.

Joint 6 has restricted limits: [-0.489, 0.262] rad — be careful when setting targets.

Historical note: an earlier `fk_arm.py` used a hand-tuned DH table (`l1=0.11875`, `l2=0.228`, `l3=0.22712`, α = `[-π/2, π/2, -π/2, π/2, -π/2, -π/2, π/2]`) referenced from `dh_param_7dof_v2.pdf`. That DH table did not match the current URDF and the script has been removed — use `fk_arm_v3.py` (URDF-driven) for anything that must agree with the visualised model.

### URDF Structure

`arm_bot.urdf.xacro` includes three files:
- `arm.xacro` — links, joints, inertial/collision/visual geometry (STL meshes from SolidWorks)
- `ros2control.xacro` — ros2_control hardware interface (`ign_ros2_control/IgnitionSystem`), position command + position/velocity state interfaces
- `gazebo.xacro` — Ignition Gazebo plugin, references `arm_robot_controllers.yaml`

`arm_moveit_config/config/arm_bot.urdf.xacro` is a separate MoveIt-generated copy — edit the original in `arm_bot/urdf/` and regenerate if needed.

After editing any URDF/Xacro file, run `colcon build --packages-select arm_bot` and re-source `install/setup.bash` — these are installed as share files and changes do not propagate until rebuilt.

C++ nodes live in `arm_bot/src/` with headers in `arm_bot/include/`; new C++ executables must be registered in `CMakeLists.txt` (`add_executable` + `install(TARGETS ...)`) and their dependencies declared in `package.xml`.

Python nodes are installed via `install(PROGRAMS ${PROJECT_NAME}/<name>.py DESTINATION lib/${PROJECT_NAME})` blocks in `CMakeLists.txt` — every new script needs its own block before `ros2 run` will find it.

### Common Pitfalls (v3 nodes)

- `fk_arm_v3.py` / `ik_arm_v3.py` need `/robot_description` to be **latched** (TRANSIENT_LOCAL durability). `robot_state_publisher` does this by default; if you launch the v3 node before `robot_state_publisher`, it sits waiting — start `rviz.launch.py` or `bot_controller.launch.py` and it will pick up the URDF.
- The default `tip_link` in `fk_arm_v3.py` / `ik_arm_v3.py` is `ee`, which matches this URDF. If you fork the URDF or rename the EE link, override with `--ros-args -p tip_link:=<name>`.

### Driving FK without Gazebo

`fk_rviz_test.launch.py` runs `joint_state_publisher` with `source_list: ['joint_commands']` — it publishes zero defaults so the robot stands upright in RViz, and merges anything published on `/joint_commands` into `/joint_states`. Drive joints from a terminal:

```bash
ros2 topic pub -r 30 /joint_commands sensor_msgs/msg/JointState "{
  name: ['joint_1','joint_2','joint_3','joint_4','joint_5','joint_6','joint_7'],
  position: [0.0, 0.3, 0.0, -0.5, 0.0, 0.1, 0.0]
}"
```

Verify the FK output matches `robot_state_publisher` (ground truth):
```bash
ros2 run tf2_ros tf2_echo base_link ee
```

## Pendant Contract (what `teach_pendant/` depends on)

The `7dof-pendant` GUI (`teach_pendant/`, pip package `pendant7dof`) imports **no**
robot node — it couples to this workspace only through (1) a ROS name/type
contract, (2) the `pendant_backend.launch.py` graph, and (3) the fact that it
`colcon build`s the **whole** workspace on launch. So "I didn't touch a pendant
node" is **not** a safe test. Before changing the robot, check it against this
contract. (Backend = `simulation only`; the `ros2_control` hardware interface is
`ign_ros2_control`/`gz_ros2_control` — there is no real-hardware driver.)

**Backend launch graph** — "Simulation ON" runs `ros2 launch arm_bot
pendant_backend.launch.py`, which starts exactly: `gazebo.launch.py` (mode
`gazebo`) **or** `arm_moveit_config demo.launch.py` (mode `moveit`),
`rviz.launch.py`, the `joint_state_broadcaster` + `arm_controller` spawners, and
`ik_arm_v3.py` + `fk_arm_v3.py` + `ik_to_trajectory.py` + `drawing_batch_planner.py`
(the last carries a tuned param block copied from `draw_and_execute_batch.launch.py`
— keep in sync). Editing any of these, or anything they include, breaks the pendant.

**Interface contract** (hardcoded in `pendant7dof/ros_bridge.py` + the launch defaults):
- Joint names `joint_1`…`joint_7`; controller named `arm_controller`; `base_link` →
  tip `ee`; `joint_6` limit `[-0.48, 0.26]` (the bridge keeps its own `JOINT_LIMITS` copy).
- Topics/types: `/arm_controller/joint_trajectory`, `/joint_states`, `/joint_commands`,
  `/ee_pose`, `/ee_target`, `drawing/strokes`, `/cartesian_path`, `/pen_canvas_norm`.

**Safe to modify freely** (pendant unaffected):
- Nodes the backend graph never starts: `slider_controller.py`, the *live*
  `drawing_trajectory_planner.py`/`drawing_executor_node`, `moveit_client.py`,
  `cartesian_path.py`, `relay_node.py`, and the C++ demos (`move_program.cpp`,
  `cartisian_move.cpp`).
- Internal logic of **any** node (incl. the four it uses) as long as published/
  subscribed topics, their types, and param names stay the same.
- Meshes/collision/inertial/colors/physics tuning — as long as joint & link **names**
  and the kinematic chain are unchanged.
- Adding new nodes/topics/launch files/packages the pendant doesn't reference.

**Breaks the pendant** (even though it's "not a pendant node"):
- Renaming any `joint_N`/link, renaming `arm_controller`, or changing its joint
  list/interfaces in `arm_robot_controllers.yaml`.
- Changing the `ee`/`base_link` names or the URDF chain (IK/FK read `/robot_description`),
  or changing joint limits (bridge clamp silently disagrees).
- Renaming/retyping any contract topic; editing `pendant_backend.launch.py` or its
  includes (`gazebo.launch.py`, controllers yaml, URDF) or the four nodes it spawns.
- **Any change that fails to build** in `arm_bot` or `arm_moveit_config` — the
  on-launch `colcon build` halts the pendant.

**Gotchas:** a joint/link/limit rename forces an edit to **pendant** files
(`ros_bridge.py` `JOINT_NAMES`/`JOINT_LIMITS`, launch `tip_link` default) anyway.
A bundled/self-contained wheel builds its **bundled** `src/` copy, not live
`~/7dof_ws` — edits need a re-`bundle` or `PENDANT7DOF_WS=~/7dof_ws` (default
non-bundled install uses the live workspace).
